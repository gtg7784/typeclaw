The reassuring part: deploying is the same thing you already do. There's no "production mode," no separate build, no special runtime. The `typeclaw start` you run on your laptop is the same command you run on a server.

## The whole deployment

Copy the agent folder to your server and start it:

```sh
ssh server
cd /opt/my-agent
typeclaw start
```

That's it. Same folder, same config, same command. The container is the unit of deployment; the server is just a different place to launch it.

What actually changes between laptop and server is small and optional: where credentials come from, whether it restarts itself after a reboot, and whether you're running more than one agent. Each is a box below — open the ones you need.

<Callout type="info" title="Why deployment is boring (and that's the goal)">
  The host side of TypeClaw is just a launcher — it starts a container and talks to it. It holds no agent state and
  loads no plugin code. So there's nothing to "drift" on a server: it's a folder and a Docker image, the same on every
  machine. The reasoning is in [How it's built](/docs/concepts/architecture).
</Callout>

## Common server needs

<Accordions>

<Accordion title="Credentials from the environment instead of files">

On a server you usually want secrets to come from your environment (or your CI's secret store) rather than the `secrets.json` file. TypeClaw checks the environment first, so just export them before starting:

```sh
export OPENAI_API_KEY=...
export SLACK_BOT_TOKEN=...
typeclaw start
```

No file edit needed — environment wins over `secrets.json`. The full resolution order is in [Secrets policy](/docs/concepts/secrets-policy). (If you run several agents that each need a _different_ key for the same provider, you can bind a custom env-var name per agent — the agent can set that up for you; see the [secrets reference](/docs/reference/secrets-json).)

</Accordion>

<Accordion title="Restart automatically after a reboot or crash">

Wrap the agent in a small **systemd unit** that runs `typeclaw start` in the agent folder, with `Restart=on-failure` and `WantedBy=multi-user.target`. That covers both cases: systemd restarts the launcher if it fails, and brings the agent back up after the host itself reboots. This is the supported path — TypeClaw deliberately runs the container without Docker's own `--restart` policy so a crashed container's logs survive for `typeclaw logs`.

</Accordion>

<Accordion title="Running several agents on one server">

Each agent is its own folder, its own container, its own state — there's no shared anything. `typeclaw compose` is a thin convenience loop over every agent folder in the current directory:

```sh
cd ~/agents/                       # holds agent-a/, agent-b/, ...
typeclaw compose start
typeclaw compose status
typeclaw compose logs -f
typeclaw compose usage --since 7d  # spend across all of them
typeclaw compose doctor
```

That's the entire model — deliberately small. More in the [compose reference](/docs/reference/compose).

</Accordion>

<Accordion title="Upgrades, rollbacks, and CI">

Each agent pins its base image to the exact `typeclaw` version in its `package.json`. To upgrade, bump that dependency and run `typeclaw start` — it regenerates and rebuilds with the new version. To roll back, downgrade the dependency and start again; old base images stay available, so the pin always finds them. Roll-forward and rollback are each one command. The release process is in [AGENTS.md](https://github.com/typeclaw/typeclaw/blob/main/AGENTS.md#release).

</Accordion>

<Accordion title="Checking on a deployment">

```sh
typeclaw status                  # running? daemon connected?
typeclaw doctor                  # check host, folder, plugins
typeclaw usage                   # token spend
typeclaw inspect <session-id>    # replay or live-tail a session
```

`doctor` is read-only by default; `doctor --fix` is safe too — every fix it makes is git-committed so you can review and revert. `doctor --json` is there for CI.

</Accordion>

</Accordions>
