A **channel** is any platform the agent talks through — Slack, Discord, Telegram, Webex, Teams, GitHub, Instagram, LINE, KakaoTalk. Same agent, many inboxes.

You already wired your **first** channel during `typeclaw init` (and, if you accepted the claim prompt it offered, you're paired as its owner too). This guide is for adding **another** one — a second platform, or your first if you chose "skip" back in `init`. The flow is the same either way.

It walks through **Slack**, the most common case; the other platforms follow the same shape, with only the sign-in details differing (covered at the bottom).

## Wire the agent into Slack

<Steps>

<Step>

### Tell TypeClaw to add Slack

From inside your agent folder:

```sh
typeclaw channel add slack-bot
```

The CLI asks for two tokens — a **bot token** (`xoxb-…`) and an **app token** (`xapp-…`). You get both from a Slack app, which you'll create in the next step. The CLI stores them safely in `secrets.json` (git-ignored, so nothing leaks into your repo) and switches the channel on.

</Step>

<Step>

### Create the Slack app

The fastest way to create a Slack app with exactly the right permissions is to paste a ready-made manifest. Go to **[api.slack.com/apps](https://api.slack.com/apps) → Create New App → From a manifest**, pick your workspace, and paste the manifest below. Then install it to your workspace.

After installing, you get the **bot token** (`xoxb-…`) under **OAuth & Permissions → Bot User OAuth Token**. The **app token** (`xapp-…`) is separate — see the box below for where to find it.

<Accordions>

<Accordion title="The manifest to paste">

```yaml
display_information:
  name: TypeClaw
oauth_config:
  scopes:
    bot:
      - app_mentions:read
      - channels:history
      - channels:read
      - chat:write
      - files:read
      - files:write
      - groups:history
      - groups:read
      - im:history
      - im:read
      - mpim:history
      - mpim:read
      - reactions:write
      - team:read
      - users:read
settings:
  event_subscriptions:
    bot_events:
      - app_mention
      - message.channels
      - message.groups
      - message.im
      - message.mpim
  interactivity:
    is_enabled: true
  socket_mode_enabled: true
```

Every scope here maps one-to-one to something the adapter actually does — `chat:write` to send, `reactions:write` to react, `files:read`/`files:write` for attachments, the `*:history`/`*:read` pairs to read channel context and members, and `users:read`/`team:read` to resolve author and workspace names.

</Accordion>

<Accordion title="Where to get the app token (xapp-…)">

The manifest only gives you the bot token. The `xapp-…` app token is a separate, app-level token you create by hand:

1. Go to **Basic Information → App-Level Tokens → Generate Token and Scopes**.
2. Add the **`connections:write`** scope (required for Socket Mode).
3. Generate it. That value is the `xapp-…` token the CLI asked for.

</Accordion>

</Accordions>

<Callout type="warn" title="Don't skip reactions:write">
  Without `reactions:write`, the agent's automatic `:eyes:` acknowledgement (and any reaction it tries to add) fails
  with Slack's `missing_scope` error — and that error shows up in your logs on every message until you fix it. If you
  already created the app without it, add it under **OAuth & Permissions → Bot Token Scopes**, then **reinstall the app
  to the workspace** so the new scope takes effect.
</Callout>

</Step>

<Step>

### Reload

```sh
typeclaw reload
```

The agent picks up the new Slack connection without a full restart. (Slack uses Socket Mode, so you don't need a public URL or any tunnel — skip ngrok and friends.)

</Step>

<Step>

### Pair yourself on the new platform

If you message the bot in Slack right now, **nothing happens. That's not a bug.**

Being owner on your _first_ channel doesn't carry over automatically — the agent recognizes you per platform, by the account you're speaking from. On a brand-new platform you're an unrecognized author, which resolves to `guest`: no permission to talk, so the message is dropped before the agent ever sees it. That's the safety floor — a stranger on any platform can't make your agent do anything until you've explicitly let them in.

So pair yourself here too:

```sh
typeclaw role claim --as owner --channel slack-bot
```

The CLI prints a one-time code. DM that code to your bot from your Slack account. The agent matches it, records you as `owner` on Slack, and from then on it answers you there.

<Accordions>

<Accordion title="Pairing across several platforms at once">

Drop the `--channel` flag and the agent listens on every wired adapter at once — handy if you want to pair the same person across Slack and Discord in one go:

```sh
typeclaw role claim --as owner
```

</Accordion>

</Accordions>

</Step>

<Step>

### Talk to it

Mention the bot in any channel it's been invited to, or DM it. It answers.

Replies in a thread stay in that thread's conversation, so the agent keeps context. A new top-level message starts a fresh conversation.

</Step>

</Steps>

<Callout type="info" title="Want to lock it down for a public channel?">
  The pairing above is enough for a personal agent in a private workspace. The moment the bot is somewhere strangers can
  reach it — like `#general` — you'll want finer control over who can do what. That's the [Lock down a public
  channel](/docs/guides/lock-down-a-public-channel) guide, and the reasoning behind it is in the [Permissions
  model](/docs/concepts/permissions-model).
</Callout>

## Other platforms

The wiring is the same shape everywhere — `typeclaw channel add <platform>`, then pair yourself. Only the sign-in changes.

```sh
typeclaw channel add discord-bot     # bot token (recommended)
typeclaw channel add discord         # QR login as your own Discord user (unofficial)
typeclaw channel add telegram-bot    # bot token
typeclaw channel add webex-bot       # bot token
typeclaw channel add webex           # email + password as your own Webex user (unofficial)
typeclaw channel add github          # PAT or GitHub App; needs a public URL
typeclaw channel add instagram       # username + password as your own Instagram user (unofficial)
typeclaw channel add line            # QR or email login, stored in secrets.json
typeclaw channel add kakaotalk       # email + password, stored encrypted
typeclaw channel add slack           # QR login as your own Slack user (unofficial)
typeclaw channel add teams           # device-code login as your own Microsoft Teams user (unofficial)
```

<Accordions>

<Accordion title="Slack — running as your own user account instead of a bot">

The main walkthrough above uses `slack-bot` (the recommended path). `typeclaw channel add slack` (no `-bot`) instead signs in as your own Slack **user** by QR code — paste the QR data URL from the Slack desktop app's "Sign in on mobile" screen. It uses the raw user session over the RTM API, so there's no app manifest or scopes to configure; the account already sees whatever channels and DMs it's a member of.

This is an **unofficial** session that can expire, and it shows no typing indicator and can't attribute reply-trigger engagement. Prefer `slack-bot` unless you specifically need to operate as a user. `slack` (user) is also the one adapter with no `channel set`/`reauth` command — to re-authenticate, remove its `channels.slack` block from both `secrets.json` and `typeclaw.json`, then run `channel add slack` again. See the [Slack (user) adapter reference](/docs/reference/slack).

</Accordion>

<Accordion title="Discord — the one extra step the portal hides">

After you paste the bot token, `channel add discord-bot` prints a `https://discord.com/oauth2/authorize?...` invite URL with the right scopes (`bot` + `applications.commands`) and a recommended permission bitfield (`permissions=277025508416`) pre-filled. That integer grants exactly what the adapter needs — Add Reactions, View Channel, Send Messages, Embed Links, Attach Files, Read Message History, Use Application Commands, Send Messages in Threads — and nothing more. Open it, pick a server, click Authorize. **The bot can't receive anything until it's in at least one server.**

To build the URL yourself, drop your application ID into:

```
https://discord.com/oauth2/authorize?client_id=YOUR_APP_ID&scope=bot+applications.commands&permissions=277025508416
```

See the [Discord adapter reference](/docs/reference/discord-bot) for the rest.

</Accordion>

<Accordion title="Discord — running as your own user account instead of a bot">

`typeclaw channel add discord` (no `-bot`) signs in as your own Discord **user** by QR code, the same way LINE does — no bot application, no server invite, no intents. It already sees whatever servers and DMs your account is in. It can't register slash commands and shows no typing indicator.

This is an **unofficial** path: Discord's Terms of Service prohibit user-account automation (self-bots), the session can need periodic re-auth, and the account is at risk of suspension. Prefer `discord-bot` unless you specifically need to operate as a user. See the [Discord (user) adapter reference](/docs/reference/discord).

</Accordion>

<Accordion title="Telegram — a bot token from BotFather">

`typeclaw channel add telegram-bot` asks for a single bot token. Create one by messaging [@BotFather](https://t.me/BotFather) on Telegram: send `/newbot`, pick a name and username, and it hands back the `123456:ABC-…` token. Telegram uses **long polling** — no public URL or tunnel needed — but only one process may poll a given token at a time, so don't run two agents against the same bot. The Bot API exposes no server-side history, so cross-channel history reads are unavailable (the agent relies on its live context buffer). See the [Telegram adapter reference](/docs/reference/telegram-bot).

</Accordion>

<Accordion title="Webex — bot token vs your own user account">

There are two Webex adapters. `typeclaw channel add webex-bot` takes a **bot token** (create one at [developer.webex.com/my-apps/new](https://developer.webex.com/my-apps/new)); it opens an outbound Mercury WebSocket, so it works behind a firewall with no public URL, and it only hears messages where the bot is @mentioned or DM'd.

`typeclaw channel add webex` (no `-bot`) signs in as your own Webex **user** with email + password. It then hears every message in every space the account belongs to — no @mention needed. The password is stored **encrypted at rest** so a host-side cron can renew the ~27h token unattended. SSO/MFA accounts aren't supported; run it only on a dedicated account. Unlike Slack and Discord — where the bot adapter is the recommended path — Webex is the reverse: the `webex` user adapter is the **recommended** default (it's the family picker's first, "(recommended)" option) because the Webex bot API only hears @mentions in group spaces, while the user account sees everything. Reach for `webex-bot` when you specifically want a bot identity and mention/DM-scoped engagement. See the [Webex (user)](/docs/reference/webex) and [Webex bot](/docs/reference/webex-bot) references.

</Accordion>

<Accordion title="GitHub — needs a public URL">

GitHub delivers events over webhooks, so it needs a public URL to reach your container. The CLI offers to create a free Cloudflare tunnel during `channel add` — that path is covered in [Expose a port](/docs/guides/first-tunnel). For the full PR-reviewer setup, see the [Code reviewer recipe](/docs/recipes/code-reviewer).

</Accordion>

<Accordion title="LINE — a personal account, not a bot">

LINE logs in as a real user account by QR or email/PIN and stores the resulting account record in `secrets.json`. It sends plain text only — no outbound attachments, stickers, rich formatting, or bot-style mentions. See the [LINE adapter reference](/docs/reference/line) for the rest.

</Accordion>

<Accordion title="Instagram — a personal account, no 2FA">

`typeclaw channel add instagram` signs in as your own Instagram user with **username + password**. Accounts protected by 2FA or a pending checkpoint challenge are rejected at login — disable 2FA on a dedicated agent account (or clear the checkpoint in a browser first). Only account metadata lands in `secrets.json`; the live session is held by the SDK under `workspace/.agent-messenger/`, so rotating credentials means replaying the login (`typeclaw channel reauth instagram`), not editing `secrets.json`. Outbound is plain text only — no attachments — and group threads engage on the agent's alias since Instagram surfaces no native @-mention. See the [Instagram adapter reference](/docs/reference/instagram).

</Accordion>

<Accordion title="KakaoTalk — credentials that renew themselves">

KakaoTalk is the only adapter TypeClaw re-authenticates for you _automatically_. Its login expires (about every 7 days) and is renewed using the email and password you provide — stored encrypted, with the key kept outside the agent folder. See [Secrets policy](/docs/concepts/secrets-policy) for the threat model. Other account-login adapters (LINE, Discord user, Instagram, Webex user, Teams) can also expire, but you refresh those yourself with `typeclaw channel reauth <adapter>`. (Webex user renews its token on a host-side cron too, and falls back to `reauth` only when that fails.)

</Accordion>

<Accordion title="Teams — device-code login as your own user">

`typeclaw channel add teams` signs in as your own Microsoft Teams user via a **device-code (AAD) login**: it asks whether the account is work/school or personal, prints a code to enter at [microsoft.com/devicelogin](https://microsoft.com/devicelogin), and stores the resulting token set in `secrets.json#channels.teams`. It then sees whatever chats and team channels the account has access to — chats **and** full team channels, not just @mentions. Both work and personal Microsoft accounts are supported.

Access tokens are short-lived (60–90 minutes). The device-code login captures an AAD refresh trio used to mint the realtime `id_token` in-container (with a REST-only fallback for send/read when it can't), but the short-lived access token itself is **not** auto-refreshed yet — rotate it by replaying the login with `typeclaw channel reauth teams` when it lapses. See the [Teams (user) adapter reference](/docs/reference/teams).

</Accordion>

</Accordions>

## Rotating credentials

If a token leaks or expires, swap it without re-adding the channel:

```sh
typeclaw channel set slack-bot      # token-based adapters — swap the saved token
typeclaw channel reauth discord     # account-login adapters — replay the interactive login
```

`channel set` covers the token-based adapters (`slack-bot`, `discord-bot`, `telegram-bot`, `webex-bot`, `github`); `channel reauth` covers the account-login ones (`line`, `instagram`, `kakaotalk`, `webex`, `discord`, `teams`). See the [channel-adapters capability table](/docs/reference/channel-adapters#cli-verbs) for the full split. The one exception is `slack` (user), which is in neither list — rotate it by removing its `channels.slack` block from `secrets.json` and `typeclaw.json`, then running `typeclaw channel add slack` again.

<Accordions>

<Accordion title="The bot won't answer — quick checks">

If you wired up Slack and the bot stays silent, the router probably dropped your message. Check:

1. `typeclaw role list` — is your account covered by any role?
2. `typeclaw logs` — there's a `denied by permissions (channel.respond)` line logged for every dropped message.
3. Did you get the workspace ID right? Slack team IDs start with `T`; Discord server IDs are numbers.

The `typeclaw role claim` flow above is the easy way to get this right without hand-typing IDs.

</Accordion>
