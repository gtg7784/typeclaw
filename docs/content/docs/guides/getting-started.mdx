By the end of this you'll have a TypeClaw agent running in its own container, listening on a websocket, and answering prompts from a terminal UI. Everything from here on out builds on this setup.

## Prerequisites

- [Bun](https://bun.sh/) ≥ 1.1
- Docker or [OrbStack](https://orbstack.dev/)

## Install

```sh
bun add -g typeclaw
```

## Scaffold an agent folder

Pick a directory. The folder name becomes the agent's name.

```sh
mkdir my-agent && cd my-agent
typeclaw init
```

`init` is an interactive wizard. It writes `typeclaw.json`, a managed `.env`, a `Dockerfile`, a `package.json` with `typeclaw` as a dependency, and a few placeholder folders (`workspace/` for free-write, `public/` for guest-visible output, `sessions/`, `.agents/skills/`, `mounts/`, `packages/`). It can also wire your first LLM provider during setup — say yes when it asks.

If you skipped the provider question (or want to add another later):

```sh
typeclaw provider add fireworks      # API key or OAuth
typeclaw model set default fireworks/kimi-k2
```

Every session uses the `default` model profile unless it asks for something more specific. See [/reference/typeclaw-json#models](/docs/reference/typeclaw-json#models) if you want fallback chains.

## Start the container

```sh
typeclaw start
```

First run builds the per-agent Docker image — a few minutes. Subsequent runs start in seconds. `start` also spawns `hostd`, a host-side daemon that handles auto port-forward and container self-restart. You won't interact with it directly; it just needs to exist.

## Talk to the agent

```sh
typeclaw tui
```

A terminal UI attaches over a websocket. Send a prompt, watch the agent think. That's the loop.

For a one-shot prompt without the UI:

```sh
typeclaw tui "hello, who are you?"
```

## What's actually running

The agent is in a Docker container called after your folder. The host CLI you just used is a launcher — `start`, `stop`, `tui`, `reload`, etc. all talk to the container over a port or to `hostd` over a Unix socket. The container has no Docker access, the host has no agent state. This split is covered in [/concepts/architecture](/docs/concepts/architecture).

## Stop / restart

```sh
typeclaw stop
typeclaw restart       # stop + start with the same flags
typeclaw logs -f       # follow container output
```

`stop` is clean — `docker stop` then `docker rm`, no leftovers in `docker ps -a`. Crashes are different: the container survives so you can read `typeclaw logs`.
