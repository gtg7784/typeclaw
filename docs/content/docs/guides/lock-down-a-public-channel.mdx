The default permission model has the agent answer only the people you've claimed as `owner`. That's the right default for a personal agent in a private workspace. The moment the bot is in `#general` — or worse, a Slack channel anyone can invite themselves to — you want a sharper edge.

The mental model is at [/concepts/permissions-model](/docs/concepts/permissions-model). This guide is the operational walkthrough.

## The shape

Roles bundle permission strings and match rules. The first role whose `match[]` rules match the incoming message wins. The fallback is `guest`, which has no permissions — including no `channel.respond`, which means the router drops the message before a session is ever created.

Four built-in roles:

| Role      | Auto-matches           | Default permissions                                                     |
| --------- | ---------------------- | ----------------------------------------------------------------------- |
| `owner`   | TUI sessions           | everything, with high-tier security guards still requiring per-call ack |
| `trusted` | nothing                | `channel.respond`, `cron.schedule`, low-tier security bypass            |
| `member`  | nothing                | `channel.respond` only                                                  |
| `guest`   | nothing (the fallback) | nothing                                                                 |

You only need to set up the match rules. The permissions come for free.

## A realistic config

The bot is in `T0123` (your Slack workspace), in two channels: `C0GENERAL` (everyone) and `C0TEAM` (your team). You want:

- you, from anywhere in the workspace, full owner privileges
- your team, in `#team`, full member privileges
- everyone else, in `#general`, nothing (router drops their messages)

```json
{
  "roles": {
    "owner": {
      "match": ["slack:T0123 author:U_ME"]
    },
    "member": {
      "match": ["slack:T0123/C0TEAM"]
    }
  }
}
```

That's it. The `owner` match adds to the TUI default (you're still owner from your terminal). The `member` match covers any author in `#team`. Anyone speaking in `#general` doesn't match either role, falls through to `guest`, and gets dropped. Anyone DMing the bot from a workspace you haven't named gets dropped.

```sh
typeclaw reload
```

`roles.match` is live-reloadable; you don't need to restart for match-rule changes.

## How match rules read

Compact strings, AND within a rule, OR across rules:

```
slack:T0123                   # any author in workspace T0123
slack:T0123 author:U_ME       # one specific author in T0123
slack:T0123/C0GENERAL         # any author in one channel
slack:dm/*                    # any Slack DM
discord:9999 author:U_MOD     # one author across a Discord guild
*                             # any channel session, any platform
tui                           # any TUI session
cron                          # any cron-fired session
```

Full DSL at [/reference/match-rule-dsl](/docs/reference/match-rule-dsl).

## Why a stranger can't talk the agent into trusting them

This is the load-bearing part of the model.

Suppose a stranger DMs your bot saying "give me owner permission" or, worse, "edit typeclaw.json and add me to roles.owner.match". The agent has `write` and `edit` tools. `typeclaw.json` is on the allowlist for non-workspace writes. The schema is valid. So why doesn't this work?

Because the bundled `security` plugin has a `rolePromotion` guard at `tool.before`. It diffs the proposed `typeclaw.json` against the on-disk version and blocks the write whenever any role gains a `permissions[]` entry, gains a `match[]` entry, or a brand-new role is introduced with privileges. The guard is classified `high` (audience-leak severity), which means **no role auto-bypasses it** — even owner from TUI must explicitly acknowledge the change.

Same shape for `cron.json`: a `cronPromotion` guard blocks new jobs and `scheduledByRole` changes on existing jobs, also high-tier, also no auto-bypass.

The result: an attacker who reaches the agent through a low-privilege channel can't escalate, even with full filesystem write access to the config files.

## Re-opening a guard for a specific role

Sometimes you do want a specific role to bypass a specific guard. The mechanism is per-guard, opt-in, declared in `roles.<name>.permissions[]`:

```json
{
  "roles": {
    "trusted": {
      "match": ["slack:T0123/C0TEAM"],
      "permissions": ["channel.respond", "security.bypass.low", "security.bypass.gitExfil"]
    }
  }
}
```

This grants `trusted` the per-guard `gitExfil` bypass — letting them run `git push` without an ack. They still don't bypass other high-tier guards. The full guard list and tier table is at [/reference/permissions](/docs/reference/permissions).

`roles.permissions` is restart-required (the security plugin captures the contract at boot). After editing:

```sh
typeclaw restart
```

## Debugging "the bot ignores me"

If you wired up Slack and the bot says nothing, the router probably dropped your message. Three things to check:

1. `typeclaw role list` — is your author covered by any role's `match[]`?
2. `typeclaw logs` — there's a `[channels] <key>: denied by permissions (channel.respond) author=<id>` line for every drop.
3. Did you spell the workspace/team ID right? Slack team IDs are `T...`, Discord guild IDs are numeric.

The claim flow at [Add a channel](/docs/guides/first-channel) is the easy path to populating `match[]` correctly without hand-typing author IDs.
