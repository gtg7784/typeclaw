"Expose a port" means two different things, and most of the time you want the easy one:

- **Reach it yourself** — a dev server the agent is running inside its container should just show up at `http://localhost:5173` on your machine. This needs **zero setup** — it already works.
- **Let the outside in** — a public `https://…` URL that GitHub's webhooks, or a friend, can reach. This is what a _tunnel_ is for, and you set it up with one command.

## Reach a server on your own machine

Nothing to do. When the agent starts a dev server inside its container — say a Vite app on port 5173 — TypeClaw automatically makes it appear at `http://localhost:5173` on your machine. Open it in your browser; the agent's server answers.

This even works for servers that only bind to "localhost" inside the container (plain `vite`, `next dev`), which a normal Docker port mapping can't reach. You don't have to do anything for it.

<Accordions>

<Accordion title="Limiting or turning off auto port-forward">

It's on by default for every port. To restrict it to a specific set, or switch it off, the agent can set the `portForward` field in `typeclaw.json` (a restart-required change):

```json
{ "portForward": { "allow": [5173, 3000] } }
```

An empty list (`{ "portForward": { "allow": [] } }`) turns it off entirely. Field details are in the [typeclaw.json reference](/docs/reference/typeclaw-json). Just ask the agent to "only forward port 5173" and it makes the change.

</Accordion>

</Accordions>

## Give it a public URL

When something _outside_ your network needs to reach the agent — GitHub delivering webhooks is the classic case — you need a public URL, because your container sits behind your router. That's a tunnel.

Tunnels are set up from the command line (this one's on you, not the agent):

```sh
typeclaw tunnel add demo --provider cloudflare-quick --upstream-port 5173
typeclaw restart
```

That gives you a free `https://…trycloudflare.com` URL — no signup, no account — pointing at port 5173 inside the container. Manage it with:

```sh
typeclaw tunnel list
typeclaw tunnel status demo
typeclaw tunnel logs demo
typeclaw tunnel remove demo
```

<Callout type="info" title="For GitHub, the channel wizard does this for you">
  If the public URL is for GitHub webhooks, you don't set the tunnel up separately — `typeclaw channel add github`
  offers to create one as part of wiring the channel, and keeps it pointed at the right place automatically even when
  the URL rotates. See [Add a channel](/docs/guides/add-a-channel) and the [Code reviewer
  recipe](/docs/recipes/code-reviewer).
</Callout>

<Callout type="warn" title="One trap: tunnel the real server, not a proxy in front of it">
  A few things (the bundled `agent-browser` dashboard is the example) run a small proxy on one port and the actual
  server on another. Point the tunnel at the **actual server's** port. Tunnel the proxy by mistake and everything looks
  fine until requests mysteriously time out.
</Callout>

## Going deeper

<Accordions>

<Accordion title="Already have your own domain / reverse proxy?">

If you run your own reverse proxy and just want TypeClaw to use a URL you control, the agent can add an `external` tunnel to `typeclaw.json` that names your URL directly — no Cloudflare subprocess, no extra binary:

```json
{
  "name": "github-webhook",
  "provider": "external",
  "for": { "kind": "channel", "name": "github" },
  "externalUrl": "https://hooks.example.com/github"
}
```

You handle the proxying upstream; TypeClaw uses the URL as-is. Full shape in the [tunnel providers reference](/docs/reference/tunnel-providers).

</Accordion>

<Accordion title="How a channel tunnel stays linked to its adapter">

A tunnel created for a channel carries a `for: { kind: 'channel', name: 'github' }` link. The adapter watches that tunnel for URL changes — Cloudflare's free tunnels get a fresh URL on every restart — and re-registers the webhook automatically, so you never chase a stale URL. This is why the GitHub wizard's tunnel "just keeps working." Reference: [tunnel providers](/docs/reference/tunnel-providers).

</Accordion>

<Accordion title="Why localhost forwarding works when Docker's -p doesn't">

A server that binds only to `127.0.0.1` inside the container isn't reachable through a normal Docker port mapping, because the bridge network doesn't route to the container's loopback. TypeClaw's host daemon runs the forwarder _inside_ the container's network namespace and relays bytes out to your machine's `localhost`, so loopback-only dev servers (Vite without `--host`, Next.js dev) show up anyway. The deeper mechanics are in [Internals → tunnels](/docs/internals/tunnels).

</Accordion>

</Accordions>
