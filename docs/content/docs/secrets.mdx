Credentials live in two gitignored files:

- **`.env`** — plain `KEY=value` lines, injected into the container via `--env-file`
- **`secrets.json`** — structured store managed by TypeClaw

## The policy

**Env-wins, file-never-auto-mutated.**

When a credential's canonical env var (e.g. `FIREWORKS_API_KEY`, `SLACK_BOT_TOKEN`) is set, that value is used at runtime. `secrets.json` is **never** auto-mutated to capture an env value, and an env value is never auto-promoted into the file.

This means:

- You can put a token in `.env` and TypeClaw will use it without rewriting `secrets.json`
- You can put a token in `secrets.json` and TypeClaw will use it when the env var is unset
- A CI pipeline can override a per-agent file value with a CI-managed env var

## The shape

Every secret-bearing field in `secrets.json` is a `Secret`:

```ts
type Secret = string | { value?: string; env?: string }
```

The string form is shorthand for `{ value }`. The object form lets you rebind a credential to a custom env-var name on demand:

```json
{
  "version": 2,
  "providers": {
    "fireworks": { "type": "api_key", "key": "fw_xxx" },
    "openai": { "type": "api_key", "key": { "env": "MY_CUSTOM_OPENAI_KEY" } }
  },
  "channels": {
    "slack-bot": {
      "botToken": { "value": "xoxb-..." },
      "appToken": { "env": "CI_SLACK_APP_TOKEN" }
    }
  }
}
```

## Resolution order

For any secret-bearing field:

1. `process.env[secret.env]` if `env` is set on the field
2. `process.env[canonicalEnvName]` (the platform's standard env var)
3. `secret.value` from `secrets.json`
4. Otherwise, the field is missing

## Encrypted blobs

Some credentials need to survive across container restarts AND be replayable for token renewal. The canonical case is the KakaoTalk renewal cron, which has to re-login with email + password every ~7 days because Kakao expires both `access_token` and `refresh_token` on the same cycle.

For those, TypeClaw uses AES-256-GCM with a per-container key stored at `~/.typeclaw/keys/<containerName>.key`. The key lives **outside the agent folder** so that an agent-folder leak (accidental `git add`, shared mount, etc.) doesn't capture both ciphertext and key.

This is defense-in-depth against agent-folder-only leaks. It does **not** protect against full host compromise — if an attacker owns your home directory, the encrypted blob and the key are both within reach.
