Part of the [channel adapters](/docs/reference/channel-adapters) family.

Transport: **Mercury WebSocket** over a user-token session (no public URL required). Setup walkthrough: [Add a channel](/docs/guides/add-a-channel).

This is the **user-account** Webex adapter (`webex`). It signs in as your own Webex user, so it receives every message in every space it belongs to — no @mention or bot invite required. For the bot-application adapter, see [Webex bot](/docs/reference/webex-bot).

<Callout type="warn" title="Use a dedicated account">
  This adapter drives a real Webex user account via email + password login. It hears every message in every space the
  account is a member of. Run it only on a dedicated agent account. SSO/MFA accounts are not supported.
</Callout>

## Auth

Email + password login. `typeclaw channel add webex` prompts for credentials and stores the resulting OAuth token set in `secrets.json#channels.webex` as one or more account records. The password is kept **encrypted at rest** (AES-256-GCM; the key lives outside the agent folder) so the host-side renewal cron can refresh tokens unattended.

Unlike bot-token adapters, `secrets.json#channels.webex` is not a flat credential object — it wraps one or more account records:

```json
{
  "channels": {
    "webex": {
      "currentAccount": "<accountId>",
      "accounts": {
        "<accountId>": {
          "account_id": "<accountId>",
          "access_token": "…",
          "refresh_token": "…",
          "expires_at": 1735689600000,
          "device_url": "…",
          "user_id": "…",
          "email": "…",
          "created_at": "…",
          "updated_at": "…"
        }
      }
    }
  }
}
```

| Field                  | Notes                                                   |
| ---------------------- | ------------------------------------------------------- |
| `currentAccount`       | `string \| null` — active account id, or `null` if none |
| `accounts.<accountId>` | Map of account id → account record (fields below)       |

Each record under `accounts.<accountId>`:

| Field               | Notes                                                            |
| ------------------- | ---------------------------------------------------------------- |
| `account_id`        | Account identifier (matches the map key)                         |
| `access_token`      | Current access token                                             |
| `refresh_token`     | Refresh token used by the renewal cron                           |
| `expires_at`        | Access-token expiry (epoch ms)                                   |
| `device_url`        | _optional_ — registered Webex device URL                         |
| `user_id`           | _optional_ — Webex person id                                     |
| `client_id`         | _optional_ — OAuth client id                                     |
| `client_secret`     | _optional_ — OAuth client secret                                 |
| `email`             | _optional_ — login email (needed for automatic renewal)          |
| `encryptedPassword` | _optional_ — AES-256-GCM envelope (needed for automatic renewal) |
| `created_at`        | Record creation timestamp                                        |
| `updated_at`        | Last-update timestamp                                            |

CLI: `typeclaw channel add webex` (interactive). **Do not use `channel set`** for credential rotation — use `typeclaw channel reauth webex` instead.

## Capabilities

| Feature                 | Supported | Notes                                                                                          |
| ----------------------- | --------- | ---------------------------------------------------------------------------------------------- |
| Outbound messages       | yes       | Sent as Webex markdown                                                                         |
| Reactions               | no        |                                                                                                |
| Message editing         | yes       | `channel_edit` maps to `PUT /messages/{id}`; own messages only                                 |
| Typing indicator        | yes       | Per-room FIFO queue preserves on-the-wire ordering                                             |
| Channel-name resolution | yes       | Resolves space titles via the Spaces API, 24h cache                                            |
| Self identity           | yes       |                                                                                                |
| History                 | yes       | Limit 100 messages; prefetch is rate-limited per room (explicit `channel_history` bypasses it) |
| Attachments             | yes       | Inbound fetch + outbound upload; text and one file post in a single multipart call             |
| Membership              | partial   | The membership API exposes no bot/human type field, so only the agent's own account is typed   |

## Token renewal

Access tokens live ~27 hours. A host-side renewal cron ticks hourly and refreshes any token within ~8 hours of expiry, then triggers a container restart through the host daemon so the new token is picked up. Renewal needs `email` + `encryptedPassword` on the account record. If renewal fails with `reauth_required`, re-authenticate with:

```sh
typeclaw channel reauth webex
```

## Quirks

- **User-account, not a bot**: receives all messages in every space the account belongs to, not just @mentions — engagement is governed by your `engagement.trigger` and match rules, not by Webex's mention model.
- **Encrypted password at rest**: the login password is stored AES-256-GCM-encrypted so the renewal cron can run unattended; the key lives outside the agent folder.
- **Per-room read rate limit**: Webex rate-limits conversation reads per room, so cold-start prefetch is admitted through a per-room limiter; explicit `channel_history` calls bypass it.
- **Reply enrichment**: Mercury delivers a parent message ref but not the parent author, so the adapter fetches the parent to attribute replies correctly.
- **Threading via parentId**: any `parentId` resolves to the thread root, so quoted-reply anchors and `thread` land in the same thread; an explicit reply anchor wins when both are present.
- **Attachment host allowlist**: inbound bearer-token fetches are restricted to HTTPS on Webex-owned hosts (`webexapis.com`, `api.ciscospark.com`, `*.webexcontent.com`) to avoid leaking the token.

## Config

```json
{
  "channels": {
    "webex": {
      "enabled": true,
      "engagement": {
        "trigger": ["mention", "reply", "dm"],
        "stickiness": { "perReply": { "window": 900000 } }
      },
      "history": {
        "prefetch": {
          "thread": { "head": 3, "tail": 10 },
          "channel": { "tail": 10 }
        }
      },
      "quotedReply": { "enabled": true, "queueDelayMs": 10000 }
    }
  }
}
```

All fields above are shared across adapters. No adapter-specific extensions exist for `webex`. Credentials live in [`secrets.json#channels.webex`](/docs/reference/secrets-json).
