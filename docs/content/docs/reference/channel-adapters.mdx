Adapters come in two shapes. **Bot adapters** authenticate as a platform bot/app and are the recommended path. **User adapters** (marked _user-account_) sign in as a real human account — useful when a platform has no bot API for your case, but unofficial and higher-maintenance.

| Adapter                                        | Auth                                                                                           | Transport         | Public URL? | Notes                                                      |
| ---------------------------------------------- | ---------------------------------------------------------------------------------------------- | ----------------- | ----------- | ---------------------------------------------------------- |
| [`slack-bot`](/docs/reference/slack-bot)       | bot token + app token                                                                          | Socket Mode       | no          | standard intent-shaped wiring                              |
| [`slack`](/docs/reference/slack)               | user token + cookie (QR login)                                                                 | RTM               | no          | user-account adapter; unofficial session                   |
| [`discord-bot`](/docs/reference/discord-bot)   | bot token                                                                                      | gateway intents   | no          |                                                            |
| [`discord`](/docs/reference/discord)           | user-token session (QR login)                                                                  | gateway           | no          | user-account adapter; against Discord ToS                  |
| [`webex-bot`](/docs/reference/webex-bot)       | bot token                                                                                      | Mercury WebSocket | no          | real-time over an outbound socket; no public URL           |
| [`webex`](/docs/reference/webex)               | user login (email + password, stored encrypted)                                                | Mercury WebSocket | no          | user-account adapter; host-side token renewal cron         |
| [`telegram-bot`](/docs/reference/telegram-bot) | bot token                                                                                      | long polling      | no          |                                                            |
| [`instagram`](/docs/reference/instagram)       | user login (username + password)                                                               | hybrid / polling  | no          | user-account adapter; plain-text, alias-only groups        |
| [`github`](/docs/reference/github)             | fine-grained PAT, or GitHub App (`appId` + `privateKey`; installations auto-resolved per repo) | webhook delivery  | yes         | pairs with a tunnel for inbound webhooks                   |
| [`line`](/docs/reference/line)                 | sub-device login (QR or email + PIN)                                                           | LINE protocol     | no          | user-account adapter; plain-text outbound only             |
| [`kakaotalk`](/docs/reference/kakaotalk)       | sub-device login (email + password, stored encrypted)                                          | LOCO protocol     | no          | survives 7-day token expiry via host-side renewal cron     |
| [`teams`](/docs/reference/teams)               | device-code login (AAD)                                                                        | trouter WebSocket | no          | user-account adapter; ~60-90min tokens, `reauth` to rotate |

## `channels.<adapter>` shape

```json
{
  "channels": {
    "slack-bot": {
      "enabled": true,
      "quotedReply": { "enabled": true, "queueDelayMs": 10000 }
    },
    "slack": { "enabled": true },
    "discord-bot": { "enabled": true },
    "discord": { "enabled": true },
    "webex-bot": { "enabled": true },
    "webex": { "enabled": true },
    "telegram-bot": { "enabled": true },
    "instagram": { "enabled": true },
    "github": {
      "enabled": true,
      "webhookPort": 8975,
      "webhookUrl": "https://hooks.example.com/github"
    },
    "line": { "enabled": true },
    "kakaotalk": { "enabled": true },
    "teams": { "enabled": true }
  }
}
```

| Field                      | Adapters | Notes                                                                                            |
| -------------------------- | -------- | ------------------------------------------------------------------------------------------------ |
| `enabled`                  | all      | live-reloadable                                                                                  |
| `quotedReply`              | all      | auto-anchor delayed replies with a `> @author: …` blockquote prefix; defaults to on, 10s         |
| `quotedReply.enabled`      | all      | set `false` to never prepend the anchor                                                          |
| `quotedReply.queueDelayMs` | all      | reply delay threshold in ms; `0` anchors every reply, large values anchor only very late replies |
| `webhookPort`              | `github` | container-side port the webhook server listens on; default `8975`                                |
| `webhookUrl`               | `github` | optional override; when omitted, the URL comes from a channel-owned tunnel                       |

Credentials live in `secrets.json#channels.<adapter>` ([/reference/secrets-json](/docs/reference/secrets-json)).

## CLI verbs

```sh
typeclaw channel add <adapter>      # interactive; all adapters
typeclaw channel set <adapter>      # rotate stored credentials
typeclaw channel reauth <adapter>   # replay the interactive login
```

`channel add` works for **every adapter** (all twelve, including `teams` via device-code login). The other two cover disjoint sets:

| Verb     | Supported adapters                                                | Notes                                                            |
| -------- | ----------------------------------------------------------------- | ---------------------------------------------------------------- |
| `set`    | `slack-bot`, `discord-bot`, `telegram-bot`, `webex-bot`, `github` | token-based adapters — swaps the saved token(s)                  |
| `reauth` | `line`, `instagram`, `kakaotalk`, `webex`, `discord`, `teams`     | account-login adapters — replays the full interactive login flow |

`slack` (user-account) is the one adapter in **neither** `set` nor `reauth` today. To re-authenticate, remove its `channels.slack` block from **both** `secrets.json` and `typeclaw.json`, then run `channel add slack` again — `channel add` refuses an adapter that is still configured in `typeclaw.json`. (`discord` user-account, by contrast, is reauthable: run `channel reauth discord` to replay the QR login in place.)

`teams` uses a device-code (AAD) login: `channel add teams` for the first sign-in and `channel reauth teams` to rotate a stale token. It is not part of `channel set` — its short-lived access token is refreshed by replaying the login (or by hand-editing `secrets.json#channels.teams`), not by pasting a token.

## Routing model

When a message arrives:

1. The adapter publishes it to the in-process message stream with a `(adapter, workspace, chat, thread, authorId)` coordinate.
2. The channel router gates the message on `channel.respond` permission ([/concepts/permissions-model](/docs/concepts/permissions-model)). No permission → silent drop.
3. A session is created (or resumed) for that coordinate.
4. Outgoing tool calls (`channel_send`, `channel_reply`, `channel_edit`) flow back through the adapter.

The router never blocks on a slow adapter. Each adapter owns its own outbound queue.

When the agent's first send of a turn lands later than `quotedReply.queueDelayMs` after the inbound was received, OR the channel had observed messages between the inbound and the reply, the router prepends a single-line `> @author: excerpt` markdown blockquote referencing the inbound so the user can see which message the reply is anchored to even after the channel has scrolled. Slack and Discord get platform-native `<@authorId>` mentions inside the quote; Telegram, LINE, KakaoTalk, and GitHub fall back to plain author names because they have no reliable id-only mention syntax inside markdown. Anchoring is first-send-only per turn — multi-part replies don't repeat the quote on every chunk.

## Cross-channel reads

Reading splits across two tools. `channel_history` fetches older messages of the **current** conversation (the chat the session is already engaged in). `channel_read` reaches a **different** chat the agent isn't necessarily in — "check the messages in #general", "what's this message id" — and folds three modes into one tool to keep the per-turn schema small:

| `channel_read` mode | Reads                                  |
| ------------------- | -------------------------------------- |
| `history`           | recent messages of an arbitrary `chat` |
| `message`           | one message by `message_id`            |
| `list`              | the channels/chats in a `workspace`    |

Not every adapter backs every mode. When an adapter hasn't registered a capability, that mode returns a `*-not-supported` error rather than failing the turn. The matrix below mirrors `ADAPTER_READ_CAPABILITIES` in `src/channels/schema.ts`, the runtime source of truth:

| Adapter        | `history` | `message` | `list` |
| -------------- | :-------: | :-------: | :----: |
| `slack-bot`    |    yes    |    yes    |  yes   |
| `discord-bot`  |    yes    |    yes    |  yes   |
| `slack`        |    yes    |    no     |   no   |
| `discord`      |    yes    |    no     |   no   |
| `webex-bot`    |    yes    |    no     |   no   |
| `webex`        |    yes    |    no     |   no   |
| `github`       |    yes    |    no     |   no   |
| `instagram`    |    yes    |    no     |   no   |
| `line`         |    yes    |    no     |   no   |
| `kakaotalk`    |    yes    |    no     |   no   |
| `teams`        |    yes    |    no     |   no   |
| `telegram-bot` |  **no**   |    no     |   no   |

Only the two first-class bot adapters (`slack-bot`, `discord-bot`) back the `message` and `list` modes. `telegram-bot` backs none — the Bot API has no server-side history read, so even `history` is unavailable there and the agent relies on its live context buffer instead.

## Message editing

`channel_edit` replaces the text of a message the agent already posted, in place. The agent passes the `message_id` a prior `channel_send` returned, plus the same `adapter`/`workspace`/`chat`, and the new `text`. It's for fixing a typo, correcting a wrong answer, or appending a result to a status message — not for continuing a conversation. The agent can only edit messages it authored; editing someone else's returns `code: "permission-denied"`.

Editing is opt-in per adapter, mirroring `ADAPTER_WRITE_CAPABILITIES` in `src/channels/schema.ts`. When an adapter hasn't registered the capability, `channel_edit` returns a `message-edit-not-supported` error rather than failing the turn; a configured-but-crashed adapter that _does_ support editing returns `message-edit-adapter-unavailable` instead, so the re-auth hint only shows when re-auth would help.

| Adapter       | `channel_edit` |
| ------------- | :------------: |
| `slack-bot`   |      yes       |
| `discord-bot` |      yes       |
| every other   |       no       |

## Per-adapter quirks

| Adapter        | Quirk                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `slack-bot`    | threads stay sticky — replies in the same thread route to the same session. Bot token needs `reactions:write` for `channel_react` and the auto-`:eyes:` engage reaction, and `chat:write` for `channel_edit`; without them Slack returns `missing_scope`. `channel_edit` maps to `chat.update` and can only edit the bot's own messages. See the [manifest in Add a channel](/docs/guides/add-a-channel) for the full scope list                                                                                                                                                                                                                                                                                                                                                     |
| `slack`        | user-account adapter — signs in as your own Slack user via QR login over the RTM API. No typing indicator; RTM events carry no parent authorship so reply-trigger engagement isn't surfaced. Unofficial session, may need re-auth. Prefer `slack-bot`. See the [Slack (user) reference](/docs/reference/slack)                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `discord-bot`  | requires `MESSAGE_CONTENT` intent enabled in the Discord developer portal; bot must be invited to a server with `bot` + `applications.commands` scopes and the recommended permission bitfield `277025508416` before it can receive anything (`channel add discord-bot` prints a ready-to-click invite URL with this value pre-filled). `channel_edit` maps to `PATCH /channels/{id}/messages/{id}` and can only edit the bot's own messages (`50005` otherwise)                                                                                                                                                                                                                                                                                                                     |
| `discord`      | user-account adapter — signs in as your own Discord user via QR login, no bot invite or intents. No typing indicator and no slash commands (user accounts can't register application commands). Self-bot automation is against Discord's ToS; prefer `discord-bot`. See the [Discord (user) reference](/docs/reference/discord)                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `webex-bot`    | real-time over an outbound Mercury WebSocket (no public URL); the bot API exposes no reactions, typing, outbound file upload, or message threading, and history has no pagination cursor — see the capabilities table above                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `webex`        | user-account adapter — signs in as your own Webex user, so it hears every message in every space (no @mention needed). Password stored encrypted; a host-side cron renews the ~27h token and restarts the container. SSO/MFA accounts unsupported. See the [Webex (user) reference](/docs/reference/webex)                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| `telegram-bot` | long polling means no public URL, but only one process can poll a given bot token at a time                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `instagram`    | user-account adapter — username/password login (2FA/checkpoint accounts rejected). Plain-text outbound only, no attachments, no typing; groups engage on alias only (no native @-mention). Scoped by raw `@instagram-dm` / `@instagram-group` workspaces. See the [Instagram reference](/docs/reference/instagram)                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| `github`       | adapter restarts on tunnel URL change and re-registers the webhook automatically. Reviews fire only on an explicit `review_requested` event — in **PAT** mode the bot is a real user requested by its login; in **GitHub App** mode the App can't be a `requested_reviewer`, so you request its **decoy user account** instead ([GitHub decoy reviewer](/docs/internals/github-decoy-reviewer)). Team-reviewer requests are honored when the bot is a member of the requested team.                                                                                                                                                                                                                                                                                                  |
| `line`         | LINE has no native @-mention or reply syntax surfaced to the adapter, so group and square engagement is alias-driven; outbound attachments are rejected because the SDK exposes text sends only                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| `kakaotalk`    | LOCO clients hold the current token in a closure; token renewal triggers a container restart via the host daemon                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `teams`        | user-account adapter — signs in as your own Microsoft Teams user via device-code login (`channel add teams` / `channel reauth teams`), so it hears chats **and** team channels alike, not just @mentions. The SDK's self-identity check returns the literal `'ME'` for every author, so the adapter suppresses its own sends via a `(chat, normalized-text)` echo fingerprint instead. Realtime rides a trouter WebSocket authed by an `id_token` minted in-container from the stored AAD refresh trio, with a REST-only fallback when it can't be minted. Access tokens live 60-90 minutes; the adapter warns on expiry but does not auto-refresh the skypetoken yet, so rotate with `channel reauth teams` when it lapses. See the [Teams (user) reference](/docs/reference/teams) |

To invite the `discord-bot` by hand, substitute your application ID into:

```
https://discord.com/oauth2/authorize?client_id=YOUR_APP_ID&scope=bot+applications.commands&permissions=277025508416
```
